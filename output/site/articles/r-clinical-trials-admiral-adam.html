結論から言うと

admiralは、Rを使って臨床試験のADaMデータセットを効率的かつ標準的に作成するための強力なパッケージです。 手作業を大幅に削減し、データ解析の品質と再現性を飛躍的に高めることができます。SASとRの連携を視野に入れつつ、現代の臨床試験プログラミングにおいて必須のスキルとなりつつあります。

admiralとは？

こんにちは、臨床統計ラボの管理人です。10年以上にわたり製薬企業やCROで臨床試験の統計解析に携わってきました。今日のテーマは、臨床試験データ解析の現場でいま最も注目されているRパッケージの一つ、admiralです。

admiralは「Analysis Data Model in R (admiral)」の略称で、R for Clinical Trialsイニシアチブの一環として開発が進められているオープンソースパッケージです。その主な目的は、CDISC ADaM（Analysis Data Model: 解析用データモデル）データセットの作成プロセスを標準化し、自動化することにあります。

CDISC ADaMデータセットは、臨床試験の統計解析の基盤となる非常に重要なデータセットです。SDTM（Standard Data Tabulation Model: 標準データ提示モデル）から解析に必要な変数を導出し、試験のデザインや解析計画書（SAP: Statistical Analysis Plan）に沿った形で構成されます。しかし、このADaMデータセットの作成は、これまでは各社・各試験で手作業や個別のSASマクロに依存することが多く、品質管理や再現性の確保に大きな課題がありました。

admiralは、この課題を解決するために、tidyverseのエコシステム上に構築されています。tidyverseとは、Rでのデータ操作を直感的かつ効率的に行うための一連のパッケージ群（dplyr, tidyr, ggplot2など）のことです。admiralはtidyverseの思想を受け継ぎ、関数型のプログラミングアプローチで、透明性が高く、再利用可能なADaM作成コードの記述を可能にします。

なぜ今、admiralが重要なのか？

  1. 効率性の向上: ADaMデータセットの作成に必要な定型的な作業を関数として提供することで、プログラミング時間を大幅に短縮します。
  2. 品質の保証: 標準化された関数を使用することで、人為的なエラーのリスクを低減し、一貫性のある高品質なデータセット作成を支援します。
  3. 再現性の確保: オープンソースであるため、コードの透明性が高く、誰でも同じ結果を再現できます。これは規制当局への提出においても非常に有利です。
  4. コミュニティによるサポート: 活発なコミュニティが存在し、機能の追加やバグ修正が継続的に行われています。疑問点があれば、コミュニティのフォーラムなどで解決策を見つけやすいのも魅力です。

統計的背景・理論

admiralの真価を理解するには、まずCDISC ADaMの基本原則を抑えておく必要があります。ADaMは、統計解析の基盤となるデータセットとして、以下の3つの主要原則に基づいています。

  1. Traceability（トレーサビリティ）: 解析結果がどのSDTMデータセットのどのデータポイントから導出されたかを明確に追跡できること。これは、規制当局がデータ監査を行う際に非常に重要です。
  2. Analysis-ready（解析準備済み）: データセットが直接解析に利用できる形式であること。つまり、解析者が追加のデータ操作を行う必要がない状態を指します。
  3. Standardized（標準化）: 変数名、値、ロジックなどがCDISCの標準に従っていること。これにより、異なる試験間での比較やデータ共有が容易になります。

admiralは、これらの原則をRの関数として実装しています。例えば、SDTMデータセット（例: DM: Demographics, VS: Vital Signs）からADaMの基幹となるADSL（Subject-Level Analysis Dataset: 被験者レベル解析データセット）を作成する際、以下のような変数を導出します。

  • 治療期に関連する日付/日数の変数: 最終投与日、来院日、治験薬投与開始日からの経過日数（ADY: Analysis Day）など。
    • ADYの計算例: $ADY = Date - Reference Date$ （ここでReference Dateは治験薬投与開始日など）
  • ベースライン値: 治験薬投与前の測定値。解析で共変量（covariate）として使われることが多いです。
  • 年齢グループ: AGE変数を基に、解析計画書で定められた年齢カテゴリ（例: <65, 65-74, $\ge 75$歳）を導出します。

これらの導出ロジックをadmiralの関数が提供することで、各社が独自に開発していたスクリプトを共通化し、ADaMの品質と作成効率を向上させます。

SAS/Rコード実装

それでは、実際にadmiralとrtablesを使ったRコードの例を見ていきましょう。今回は、架空のSDTMデータからADSL（被験者レベル解析データセット）の一部を作成し、簡単な要約表を作成する例を示します。

Rコード（admiral & rtables）

まず、必要なパッケージをロードし、ダミーのSDTMデータ（DM, EX, VS, LB）を作成します。実際の臨床試験では、これらのデータはトランスポートファイル（XPT）から読み込まれます。

# パッケージのロード
library(tidyverse)
library(admiral)
library(rtables)
library(tern)

# 1. ダミーSDTMデータの作成
# DM (Demographics) データ
dm <- tibble::tribble(
  ~STUDYID, ~USUBJID, ~SITEID, ~SEX,   ~AGE, ~RACE,           ~ARM,            ~ACTARMCD, ~TRT01P,   ~TRT01A,   ~RFSTDTC,   ~RFENDTC,

  "STUDY01", "001-004", "102",   "F",    68,   "WHITE",         "Placebo",         "PLC",     "Placebo", "Placebo", "2023-01-15", "2023-04-14",
  "STUDY01", "001-005", "103",   "M",    71,   "ASIAN",         "Drug X 10mg",     "DRG10",   "Drug X 10mg", "Drug X 10mg", "2023-01-20", "2023-04-19",
  "STUDY01", "001-006", "103",   "F",    64,   "WHITE",         "Drug X 20mg",     "DRG20",   "Drug X 20mg", "Drug X 20mg", "2023-01-25", "2023-04-24"
) %>%
  mutate(
    RFSTDTC = as.Date(RFSTDTC),
    RFENDTC = as.Date(RFENDTC)
  )

# EX (Exposure) データ - 簡略化
ex <- tibble::tribble(
  ~STUDYID, ~USUBJID, ~EXSTDTC,
  "STUDY01", "001-001", "2023-01-01",
  "STUDY01", "001-002", "2023-01-05",
  "STUDY01", "001-003", "2023-01-10",
  "STUDY01", "001-004", "2023-01-15",
  "STUDY01", "001-005", "2023-01-20",
  "STUDY01", "001-006", "2023-01-25"
) %>%
  mutate(EXSTDTC = as.Date(EXSTDTC))

# VS (Vital Signs) データ - 簡略化
vs <- tibble::tribble(
  ~STUDYID, ~USUBJID, ~VSTESTCD, ~VSSTRESN, ~VSDTC,
  "STUDY01", "001-001", "WEIGHT",    75.0,      "2023-01-01",
  "STUDY01", "001-001", "WEIGHT",    74.5,      "2023-02-01",
  "STUDY01", "001-002", "WEIGHT",    60.0,      "2023-01-05",

  "STUDY01", "001-004", "WEIGHT",    65.0,      "2023-01-15",

  "STUDY01", "001-006", "WEIGHT",    62.0,      "2023-01-25"
) %>%
  mutate(VSDTC = as.Date(VSDTC))

# LB (Laboratory Test) データ - 簡略化
lb <- tibble::tribble(
  ~STUDYID, ~USUBJID, ~LBTESTCD, ~LBSTRESN, ~LBNRIND, ~LBDTC,
  "STUDY01", "001-001", "ALT",       25.0,      "NORMAL", "2023-01-01",
  "STUDY01", "001-001", "ALT",       35.0,      "HIGH",   "2023-02-01",
  "STUDY01", "001-002", "ALT",       30.0,      "NORMAL", "2023-01-05",
  "STUDY01", "001-003", "ALT",       40.0,      "HIGH",   "2023-01-10",
  "STUDY01", "001-004", "ALT",       28.0,      "NORMAL", "2023-01-15",
  "STUDY01", "001-005", "ALT",       32.0,      "NORMAL", "2023-01-20",
  "STUDY01", "001-006", "ALT",       29.0,      "NORMAL", "2023-01-25"

  mutate(LBDTC = as.Date(LBDTC))

# 2. ADSLのベースをDMから作成
adsl <- dm %>%
  # `TRT01A`, `TRT01P` の値を変換 (必要に応じて)
  mutate(TRT01A = as.factor(TRT01A), TRT01P = as.factor(TRT01P))

# 3. `admiral`関数で新しい変数を導出
adsl <- adsl %>%
  # 年齢グループ (AGEGR1) を導出
  # `cut_value()` は `admiral` のヘルパー関数で、カテゴリ分けに便利です。
  derive_var_age_gr(
    age_var = AGE,
    new_var = AGEGR1,
    intervals = c(0, 65, 75),
    labels = c("<65", "65-74", ">75")
  ) %>%
  # 治験薬投与開始日 (TRTSDTM) からの経過日数 (ADY) と週 (AWEEK) を導出
  # EXデータセットのEXSTDTCを治療開始日として使用
  derive_vars_study_start_dt(
    dataset_ex = ex, 
    ex_vars = exprs(EXSTDTC)
  ) %>%
  # `derive_vars_dt()` は日付変数から日付、月、年などを導出します。
  # ADSLの場合、RFSTDTCなどをベースにしても良いですが、ここではEXSTDTCをADaMの治療開始日と見立てます
  # `derive_vars_dy()` で日数、`derive_vars_week_ext()` で週数を導出
  derive_vars_dy(
    dataset = .,
    reference_date = RFSTDTC, # 基準となる日付
    source_vars = exprs(RFSTDTC, RFENDTC)
  ) %>%
  derive_vars_aday(dataset = .,
                   date = RFSTDTC, # 基準となる日付
                   base_date = RFSTDTC, # 解析開始日からの日数計算のベース
                   new_var = ADY)

# 必要に応じて、他のSDTMデータからの変数マージや導出
# 例: ベースライン体重 (BMIBL) やベースラインALT異常値 (LBHIGHN)
# ここでは簡略化のため、VSとLBから最初の測定値をベースラインとみなします。
# `derive_vars_merged()` は複数のデータセットを結合します。
# まず、ベースラインのVSとLBを取得します。
vs_bl <- vs %>%
  filter(VSTESTCD == "WEIGHT") %>%
  arrange(USUBJID, VSDTC) %>%
  group_by(USUBJID) %>%
  slice(1) %>%
  ungroup() %>%
  select(USUBJID, VSSTRESN, VSDTC) %>%
  rename(BMIBL = VSSTRESN, VSDTCL = VSDTC)

lb_bl <- lb %>%
  filter(LBTESTCD == "ALT") %>%
  arrange(USUBJID, LBDTC) %>%
  group_by(USUBJID) %>%
  slice(1) %>%
  ungroup() %>%
  select(USUBJID, LBNRIND, LBDTC) %>%
  rename(ALTBLIND = LBNRIND, LBDTCL = LBDTC)

# ADSLにベースライン情報をマージ
adsl <- adsl %>%
  derive_vars_merged(
    dataset_add = vs_bl,
    by_vars = exprs(STUDYID, USUBJID),
    new_vars = exprs(BMIBL, VSDTCL)
  ) %>%
  derive_vars_merged(
    dataset_add = lb_bl,
    by_vars = exprs(STUDYID, USUBJID),
    new_vars = exprs(ALTBLIND, LBDTCL)
  )

# 異常値インジケーターの導出 (例: ベースラインALTがHIGHなら1, それ以外は0)
adsl <- adsl %>%
  mutate(LBHIGHN = case_when(ALTBLIND == "HIGH" ~ 1, TRUE ~ 0))

# ADSLデータセットの確認
print(adsl)

# 4. `rtables`と`tern`を使った人口統計学的な要約表の作成
# まず、解析用にデータセットを調整
# ternパッケージの`add_count_fct()`を使って、factorレベルごとのカウントを追加
# これは後でrtablesで表示するための準備です。

# 例えば、AGEGR1とSEXの分布を治療群ごとに集計する
lyt <- basic_table() %>%
  split_cols_by("TRT01A") %>%
  add_col_totals() %>%
  analyze("AGE", 
          fns = list(N = ~sum(!is.na(.x)),
                     Mean = ~mean(.x, na.rm = TRUE),
                     SD = ~sd(.x, na.rm = TRUE),
                     Median = ~median(.x, na.rm = TRUE),
                     Min = ~min(.x, na.rm = TRUE),
                     Max = ~max(.x, na.rm = TRUE)
                     ) 
          ) %>%
  # カテゴリカル変数の集計
  split_rows_by("SEX", split_fun = add_overall_level("All")) %>%
  summarize_row_groups()

# テーブルの生成
tbl <- build_table(lyt, adsl)

# テーブルの表示
tbl

コード解説: * ステップ1: tibble::tribble() を使って、SDTMのデモグラフィック（DM）、投与（EX）、バイタルサイン（VS）、臨床検査値（LB）データセットの簡略版を作成しました。実際の試験では、SAS XPT形式のCDISCデータが入力となります。 * ステップ2: dmデータセットを基に、ADSLの初期段階を作成します。 * ステップ3: admiralの主要関数を使って、解析に必要な新しい変数を導出しています。 * derive_var_age_gr(): AGE変数から年齢グループAGEGR1を定義します。区間とラベルを引数として渡すことで、簡単にカテゴリ化できます。 * derive_vars_study_start_dt(): EXデータセットの投与開始日を抽出し、主たる治療開始日とします。 * derive_vars_dy(): リファレンス日付から日数を導出します。 * derive_vars_aday(): 解析開始日（RFSTDTC）からの経過日数 ADY を導出します。 * derive_vars_merged(): VSやLBなどの他のSDTMデータセットから、ベースライン情報（例: BMIBLやALTBLIND）をUSUBJID（被験者識別子）をキーとしてマージします。これはADaM作成において非常に一般的な操作です。 * mutate() と case_when(): tidyverseの関数を使い、条件に基づいて新しい変数（例: LBHIGHN）を作成します。admiralはtidyverseとシームレスに連携します。 * ステップ4: rtablesパッケージとternパッケージを使用して、作成したadslデータセットから人口統計学的な要約表を生成しています。rtablesは、臨床試験報告書で求められるような複雑なテーブルを柔軟に作成できる点が特徴です。

/* 2. ADSLのベースをDMから作成 */
data adsl_sas;
  set dm;
  TRTSDTM = EXSTDTC; /* 仮にEXSTDTCを治療開始日とする */
  format TRTSDTM yymmddd10.;
run;

/* 3. 新しい変数の導出 */
/* 年齢グループ (AGEGR1) を導出 */
proc format;
  value agefmt low-<65 = '<65'
               65-74   = '65-74'
               75-high = '>75';
quit;

data adsl_sas;
  set adsl_sas;
  if AGE < 65 then AGEGR1 = '<65';
  else if AGE >= 65 and AGE <= 74 then AGEGR1 = '65-74';
  else if AGE > 74 then AGEGR1 = '>75';
  format AGEGR1 $9.;

  /* 解析開始日からの経過日数 (ADY) を導出 */
  ADY = RFSTDTC - RFSTDTC; /* この例ではRFSTDTCをベースにしているので常に0。実際は異なるベース日付 */

  /* ベースライン体重 (BMIBL) とALT異常値 (LBHIGHN) をマージ */
  /* ベースラインVS */
  proc sort data=vs out=vs_sorted;
    by USUBJID VSDTC;
  run;
  data vs_bl;
    set vs_sorted;
    by USUBJID;
    if first.USUBJID and VSTESTCD = 'WEIGHT';
    rename VSSTRESN = BMIBL VSDTC = VSDTCL;
    keep STUDYID USUBJID BMIBL VSDTCL;
  run;

  /* ベースラインLB */
  proc sort data=lb out=lb_sorted;
    by USUBJID LBDTC;
  run;
  data lb_bl;
    set lb_sorted;
    by USUBJID;
    if first.USUBJID and LBTESTCD = 'ALT';
    rename LBNRIND = ALTBLIND LBDTC = LBDTCL;
    keep STUDYID USUBJID ALTBLIND LBDTCL;
  run;

proc sql;
  create table adsl_sas as
  select
    t1.*,
    t2.BMIBL,
    t2.VSDTCL,
    t3.ALTBLIND,
    t3.LBDTCL
  from adsl_sas t1
  left join vs_bl t2
    on t1.USUBJID = t2.USUBJID
  left join lb_bl t3
    on t1.USUBJID = t3.USUBJID
  ;
quit;

data adsl_sas;
  set adsl_sas;
  /* 異常値インジケーターの導出 */
  LBHIGHN = (ALTBLIND = 'HIGH'); /* HIGHなら1, それ以外は0 */
run;

/* ADSLデータセットの確認 */
proc print data=adsl_sas(obs=5);
  var USUBJID AGE AGEGR1 RFSTDTC ADY BMIBL LBHIGHN;
run;

/* 4. SASでの要約表の作成例 (PROC FREQ) */
proc freq data=adsl_sas;
  tables TRT01A * (SEX AGEGR1);
run;

SASコード解説: * ステップ1: dataステップでダミーのSDTMデータを手動で作成しています。 * ステップ2: dmデータセットを基にadsl_sasの初期段階を作成します。 * ステップ3: DATAステップとPROC FORMATでAGEGR1を導出し、PROC SQLとDATAステップで他のデータセットからのマージや異常値インジケーターの導出を行っています。admiralの関数とは異なり、各処理が分離しており、コードの可読性や保守性が劣る可能性があります。 * ステップ4: PROC FREQを使ってカテゴリカル変数の単純な頻度集計を行っています。より複雑な統計表にはPROC REPORTやPROC TABULATEなどを使用しますが、その記述はさらに複雑になります。

これらのコード例から、admiralが提供する関数群が、いかに効率的で読みやすいコード記述を可能にするかお分かりいただけるかと思います。特にderive_var_age_gr()のような関数は、SASでのPROC FORMATとif-elseロジックの組み合わせに相当し、簡潔に記述できます。{{internal_link:SASからRへの移行ガイド}}

実務での活用場面

admiralは、臨床試験のデータサイエンスとプログラミングのワークフローにおいて、多岐にわたる場面で活用できます。

  • ADaMデータセットの初回作成: 治験の開始時やプロトコル承認後、SDTMデータが利用可能になった段階で、解析計画書に基づいてADaMデータセットを一から作成する際に最も威力を発揮します。
  • ADaMデータセットの更新・改訂: プロトコル改訂や解析計画書の変更により、既存のADaMデータセットに修正が必要になった場合、admiralのモジュール化された関数を使用することで、影響範囲を最小限に抑えつつ、効率的に変更を適用できます。
  • 品質管理（QC）とバリデーション: admiralで作成されたコードは透明性が高く、各変数の導出ロジックが明確なため、データマネジメントや統計部門によるQC（Quality Control: 品質管理）やバリデーション（Validation: 妥当性確認）のプロセスを効率化します。標準化された関数を使用することで、バリデーションの負担も軽減されます。
  • トレーサビリティの確保: admiralの関数は、SDTMからADaMへのデータの流れを明確にするように設計されています。これは、規制当局（例: FDA, PMDA）へのデータ提出時に求められるトレーサビリティの要件を満たす上で非常に重要です。
  • グローバル標準の導入: 複数のCROや異なる地域のチームと連携するグローバルスタディにおいて、admiralを導入することで、ADaM作成プロセスを統一し、データの一貫性と品質を確保することができます。

海外の最新動向

admiralを含むR for Clinical Trialsエコシステムは、海外の主要な統計コミュニティや規制当局でも大きな注目を集めています。

R for Clinical Trials Working Group

R Consortiumが主催するR for Clinical Trials Working Groupは、臨床試験でのRの活用を推進するための重要な役割を担っています。PHUSE（Pharmaceutical Users Software Exchange）、CDISC（Clinical Data Interchange Standards Consortium）、FDA（米国食品医薬品局）の専門家も参加し、Rパッケージの開発ガイドラインやバリデーション戦略について議論を重ねています。admiralは、このワーキンググループが推奨する標準化されたパッケージ群の筆頭として位置づけられています。

PSI, ASA, PHUSE, CDISCイベントでの発表

PSI（Statisticians in the Pharmaceutical Industry）、ASA（American Statistical Association）、PHUSE、CDISCといった国際的なカンファレンスでは、admiralやrtables、tern、TplyrなどのClinical Trial用Rパッケージに関する発表が頻繁に行われています。特に、SDTMからADaMへの変換の自動化、解析結果のテーブル・リスト・フィギュア（TLF）作成の効率化、そしてバリデーションプロセスの改善に関する事例発表が増えています。

AIプログラミングとadmiralの連携

最近では、ChatGPTやGitHub CopilotのようなAIプログラミングツールが注目を集めています。これらのツールは、自然言語での指示に基づいてコードを生成したり、既存のコードを補完したりすることができます。admiralのような標準化されたパッケージと組み合わせることで、ADaM作成の初稿コードをAIが生成し、人間がそれをレビュー・修正するというワークフローが現実的になりつつあります。これにより、定型的なコーディング作業の時間がさらに短縮され、プログラマーはより複雑なロジック設計や品質管理に集中できるようになるでしょう。{{internal_link:AI時代の臨床統計プログラミング}}

規制当局の動向

FDAはRを正式な統計解析ソフトウェアとして認める方向にあり、Rで作成された解析結果の提出を積極的に受け入れています。PMDA（医薬品医療機器総合機構）も同様に、オープンソースソフトウェアの活用に前向きな姿勢を示しており、admiralのような透明性の高いパッケージは、将来的な規制当局への提出において、より広く受け入れられる可能性を秘めています。

よくある質問（FAQ）

Q1: admiralを使うメリットは何ですか？

A1: admiralは、ADaMデータセットの作成プロセスを標準化し、効率性と品質を大幅に向上させます。具体的には、定型的な導出ロジックが関数として提供されるため、プログラミング時間の短縮、人為的エラーの削減、コードの再現性・透明性の確保に貢献します。これにより、データ解析の信頼性が高まります。

Q2: admiralはSASを完全に置き換えるものですか？

A2: 現時点では、admiralがSASを完全に置き換えるものではありません。多くの製薬企業やCROでは、依然としてSASが主要なプログラミングツールとして使われています。しかし、admiralはSASで課題とされてきたADaM作成の標準化と効率化において強力な代替手段・補完ツールとなり、RとSASの共存・連携が今後の主流となるでしょう。例えば、SDTMデータはSASで作成し、ADaMはadmiralで作成するといったワークフローも考えられます。

Q3: admiralはADaM以外のデータセット作成にも使えますか？

A3: admiralはCDISC ADaMデータセットの作成に特化して設計されています。そのため、他の標準（例: SDTM）や社内独自のデータセットの作成には直接適用できません。ただし、admiralが採用しているtidyverseのデータ操作の考え方や、モジュール化された関数設計は、他のデータセット作成にも応用できる良いプラクティスとなります。

Q4: admiralの学習リソースはどこにありますか？

A4: admiralの公式ウェブサイト（https://pharmaverse.github.io/admiral/）には、詳細なドキュメント、チュートリアル、利用例が豊富に用意されています。また、GitHubリポジトリにはソースコードとIssueトラッカーがあり、コミュニティのメーリングリストやR for Clinical Trials Working Groupの活動も活発です。これらのリソースを活用することで、効率的に学習を進めることができます。

Q5: AIプログラミングとadmiralの連携は可能ですか？

A5: はい、非常に有望な連携です。ChatGPTやGitHub CopilotなどのAIプログラミングツールは、自然言語での指示に基づいてadmiralのコードの初稿を生成したり、コード補完を支援したりすることができます。これにより、定型的なコーディング作業の負荷を軽減し、プログラマーはより複雑なロジック設計や品質管理に注力できるようになります。将来的には、AIが解析計画書から直接ADaMコードを生成するような高度な連携も期待されます。
