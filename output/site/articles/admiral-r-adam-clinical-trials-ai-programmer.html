結論から言うと

admiral Rパッケージは、臨床試験におけるADaMデータセットの標準化・自動化を強力に推進します。 これにより、生物統計プログラマーは作業効率を大幅に向上させ、解析の再現性と品質を確保できます。 SAS中心のワークフローからRへの移行を検討している方にとって、admiralはAI時代のデータハンドリングを支える鍵となるでしょう。

admiralとは？

admiralは、Rパッケージエコシステムの一部であり、特に臨床試験の文脈でADaM（Analysis Data Model）データセットを作成するために設計されています。ADaMは、FDA（米国食品医薬品局）やPMDA（医薬品医療機器総合機構）などの規制当局への申請に必要な、標準化された解析用データセットの構造です。

従来のADaM作成は、複雑なSASマクロや手作業に頼ることが多く、時間と労力がかかり、エラーのリスクも伴いました。しかし、admiralは、tidyverseの思想に基づいて開発され、一貫性のある記述と高い可読性でADaM作成プロセスを簡素化します。これは、pharmaverseと呼ばれる製薬業界向けのRパッケージ群の中核をなすもので、データサイエンスの力で臨床開発を加速させることを目指しています。

admiralの主な目的は、CDISC ADaM IG（Implementation Guide）に準拠したADaMデータセットの作成を支援することです。具体的には、以下のような機能を提供します。

  • ヘルパー関数: ADSL（Subject-Level Analysis Dataset）やADPC（Pharmacokinetics Concentration Analysis Dataset）などの標準ADaMデータセットの変数を導出する関数を提供します。
  • トレーサビリティ（Traceability）: 変数の導出過程を明確にし、ソースデータから最終的なADaMデータセットまで、データの流れを追跡しやすくします。これは規制当局への説明責任を果たす上で非常に重要です。
  • 再利用性: モジュール化された関数により、異なる試験間でコードの再利用が可能となり、開発効率が向上します。

統計的背景・理論

admiralは直接的に高度な統計理論を実装するものではありませんが、その背後には臨床試験のデータサイエンスにおける品質と再現性という重要な原則があります。 統計解析を行う前のデータ準備段階で、データの正確性、一貫性、そして解析への適合性を保証することは、統計的推論の妥当性に直結します。

ADaMデータセットが求められるのは、以下の統計的・規制的要件を満たすためです。

  1. 標準化: CDISC（Clinical Data Interchange Standards Consortium）によって定められたADaMは、解析結果の比較可能性と規制当局によるレビューの効率化を目的としています。例えば、ADSLにおける年齢変数（AGE）は、特定のルールに基づいて導出されることが推奨されています。
  2. トレーサビリティ（Traceability）: 解析用変数が、どのSDTM（Study Data Tabulation Model）変数から、どのようなロジックで導出されたかを明確にすることです。これは、$Y = f(X_1, X_2, \dots, X_n)$ のような関数関係で表現されることが多いですが、admiralの関数は、この$f$を記述的かつ検証可能な形で提供します。これにより、解析の透明性が高まり、後続の統計解析（例: 共変量調整モデル）の信頼性が向上します。
  3. 再現性（Reproducibility）: 同じデータとコードを用いれば、常に同じADaMデータセットが生成されることを保証します。これは、現代のデータサイエンスにおいて研究の信頼性を担保する上で不可欠です。

admiralは、これらの原則を具現化し、生物統計プログラマーが解析の基盤となる高品質なデータセットを効率的に構築するための強力なツールです。解析に先立つデータ準備の品質を高めることは、最終的な統計解析結果の頑健性（Robustness）と解釈可能性を高めることに他なりません。

SAS/Rコード実装

それでは、admiralとSASを比較しながら、具体的なADaMデータセットの作成例を見ていきましょう。 ここでは、ADSLデータセットの簡単なデモとして、年齢（AGE）と年齢群（AGEGR1）を導出するケースを考えます。

まずは、サンプルデータを作成します。

サンプルデータ生成（R）

library(tidyverse)
library(lubridate) # 日付操作に便利
library(admiral)   # ADaMデータセット作成の中核パッケージ

# サンプル元のSDTM-likeデータを作成
adsl_raw <- tribble(
  ~STUDYID, ~USUBJID, ~BRTHDT,       ~RFENDT,
  "STUDY01", "SUBJ001", "1980-05-15", "2023-01-01",
  "STUDY01", "SUBJ002", "1992-11-20", "2023-01-01",
  "STUDY01", "SUBJ003", "1975-03-01", "2023-01-01",
  "STUDY01", "SUBJ004", "2000-07-25", "2023-01-01"
) %>%
  mutate(
    BRTHDT = as.Date(BRTHDT), # 誕生日を日付型に変換
    RFENDT = as.Date(RFENDT)  # 参考終了日を日付型に変換
  )

print(adsl_raw)

RコードによるADaM作成（admiral）

このサンプルデータから、admiralを使って年齢（AGE）とその年齢群（AGEGR1）を導出するADSLデータセットを作成します。 admiralは、日付変数を数値に変換するヘルパー関数や、年齢を計算する関数を提供しています。

# admiralでの年齢計算には、日付をISO 8601フォーマットまたは数値表現に変換することが推奨されます。
# ここでは、簡略化のためRのDate型をそのまま利用し、derive_var_age_years()に渡します。
# admiralのderive_var_age_years()は、内部で日付計算を適切に処理します。

adsl <- adsl_raw %>%
  # derive_vars_dtm() はSDTMのDTC変数から日時変数を作成する際に便利ですが、
  # この例ではBRTHDTとRFENDTがDate型なので直接利用します。
  derive_var_age_years(
    new_var = AGE, # AGE変数を新規作成
    date_var = RFENDT, # どの時点の年齢かを示す日付変数
    age_var = BRTHDT # 誕生日変数
  ) %>%
  # 年齢群 (AGEGR1) を作成
  mutate(
    AGEGR1 = case_when(
      AGE < 20 ~ "<20",
      AGE >= 20 & AGE < 40 ~ "20-39",
      AGE >= 40 & AGE < 60 ~ "40-59",
      TRUE ~ ">=60"
    )
  ) %>%
  # 最終的なADSL形式に整えるため、選択する変数を明示
  select(STUDYID, USUBJID, BRTHDT, RFENDT, AGE, AGEGR1)

print(adsl)

上記のコードでは、admiralのderive_var_age_years()関数を使って、誕生日（BRTHDT）と参考終了日（RFENDT）から年齢（AGE）を計算しています。 その後、dplyr::mutate()関数で年齢群（AGEGR1）を作成しています。 admiralの関数はパイプ演算子（%>%）と親和性が高く、データの流れを直感的に記述できるのが特徴です。

SASコードによるADaM作成（比較）

次に、同じ処理をSASで行う場合のコードを見てみましょう。

/* サンプルデータ生成 (SAS) */
data adsl_raw_sas;
  informat BRTHDT RFENDT date9.;
  format BRTHDT RFENDT date9.;
  input STUDYID $ USUBJID $ BRTHDT RFENDT;
  datalines;
STUDY01 SUBJ001 15MAY1980 01JAN2023
STUDY01 SUBJ002 20NOV1992 01JAN2023
STUDY01 SUBJ003 01MAR1975 01JAN2023
STUDY01 SUBJ004 25JUL2000 01JAN2023
;
run;

/* ADSLデータセット作成 (SAS) */
data adsl_sas;
  set adsl_raw_sas;

  /* 年齢 (AGE) の導出 */
  /* INTCK関数で年数を計算し、誕生日が参照日よりも後かどうかで調整 */
  AGE = int(intck('YEAR', BRTHDT, RFENDT) - (month(BRTHDT)*100 + day(BRTHDT) > month(RFENDT)*100 + day(RFENDT)));

  /* 年齢群 (AGEGR1) の導出 */
  if AGE < 20 then AGEGR1 = '<20';
  else if AGE >= 20 and AGE < 40 then AGEGR1 = '20-39';
  else if AGE >= 40 and AGE < 60 then AGEGR1 = '40-59';
  else AGEGR1 = '>=60';

  label AGE = 'Age in Years';
  label AGEGR1 = 'Pooled Age Group';
run;

proc print data=adsl_sas;
  var STUDYID USUBJID BRTHDT RFENDT AGE AGEGR1;
  format BRTHDT RFENDT yymmdd10.; /* 表示形式の指定 */
run;

SASコードでは、intck関数を用いた年齢計算や、if-then-else構文による年齢群の作成が行われています。Rのadmiralと比較すると、SASでは日付計算のロジックを自力で記述する必要がある場面が多いことがわかります。admiralの関数は、このような標準的な導出ロジックをカプセル化（Encapsulation）しているため、より簡潔でエラーの少ないコードを書くことが可能です。

rtablesとの連携例

admiralで作成したADaMデータセットは、rtablesなどのパッケージと組み合わせることで、そのまま治験総括報告書（CSR）向けのテーブル（表）を生成できます。{{internal_link:rtablesの使い方}}について詳しくはこちらもご覧ください。

library(rtables) # 臨床試験テーブル作成パッケージ

# ADSLデータセットから年齢層別の被験者数を集計するテーブルを作成
table_adsl <- build_table(
  df = adsl, # admiralで作成したADSLデータセット
  # テーブル構造を定義
  ## 行の定義: AGEGR1（年齢群）を行として、各年齢群のNを計算
  rtables::r_align(rtables::r_grp_strata(col = "AGEGR1")) %>%
  rtables::r_count(col = "AGEGR1", label = "被験者数N")
)

# 生成されたテーブルを表示
print(table_adsl)

Table 1.1: 被験者数N

           AGEGR1
           <20   20-39 40-59 >=60  Total
           ------ ------ ------ ------ ------
 被験者数N       0      2      2      0      4

このように、admiralで整備されたADaMデータセットは、rtablesのような専用のパッケージと組み合わせることで、データ解析から報告書作成までの一連のワークフローをR上で完結させることができ、生物統計プログラマーの生産性を大幅に向上させます。

実務での活用場面

admiralは、臨床試験の様々なフェーズでその真価を発揮します。

  1. プロトコル作成・統計解析計画書（SAP）作成フェーズ: プロトコルやSAPで定義された解析変数の導出ロジックを、admiralの関数を用いて具体的なRコードとして早期に検討できます。これにより、仕様の曖昧さをなくし、統計家とプログラマー間の認識齟齬を防ぎます。
  2. ADaMデータセット作成フェーズ: CDISC ADaM IGに準拠したADaMデータセットを効率的かつ標準化された方法で作成します。特に、ADSL（被験者レベル解析データセット）、ADPC（薬物濃度解析データセット）、ADEG（心電図解析データセット）など、頻繁に利用されるADaMの導出に強みを発揮します。{{internal_link:ADaMデータセットの基本}}についてもっと詳しく知りたい方はこちら。
  3. 解析・報告フェーズ: admiralで作成されたADaMは、rtablesやTplyr、ternといった他のpharmaverseパッケージと連携し、治験総括報告書（CSR）向けのテーブル、リスト、フィギュア（TLF）を自動生成するための基盤となります。これにより、手作業によるミスを減らし、報告書の作成期間を短縮できます。
  4. 承認申請フェーズ: 規制当局へのデータ提出において、admiralが提供するトレーサビリティは非常に重要です。データリネージ（Data Lineage）が明確に記述されたコードは、データ提出時の透明性を高め、スムーズなレビューに貢献します。

AI時代において、ルーティンワークの自動化は必須です。admiralは、ADaM作成という反復的かつ規則的な作業を自動化することで、生物統計プログラマーがより高度な統計解析や新たな手法の検討といった、付加価値の高い業務に集中できる環境を提供します。

海外の最新動向

admiralを含むpharmaverseプロジェクトは、PSI（Statisticians in the Pharmaceutical Industry）、ASA（American Statistical Association）、PHUSE（Pharmaceutical Users Software Exchange）といった主要な国際的なコミュニティで活発に議論され、その普及が進んでいます。

  • PHUSE CSS（Computational Science Symposium）: 毎年開催されるPHUSEのCSSでは、pharmaverseに関するセッションが数多く設けられ、新しいパッケージの紹介、機能拡張の議論、そして実務適用例の共有が行われています。
  • PSI / ASA: これらの学会では、Rを用いた臨床試験データ解析のベストプラクティスに関する発表が増加しており、admiralがADaM作成の「デファクトスタンダード」となりつつあることが示唆されています。特に、FDAもRの利用を積極的に推奨しており、Rにおけるデータ提出の準備が着実に進んでいます。
    • FDAによるRの利用促進に関する情報: https://www.fda.gov/media/132711/download

海外の主要な製薬企業やCROでは、SASとRのハイブリッドなワークフローが一般的になりつつあり、新規プロジェクトではRが積極的に採用されるケースも増えています。 これは、Rの持つオープンソースとしての柔軟性、最新統計手法への対応速度、そして強力なコミュニティサポートが評価されているためです。

AI技術の進化は、統計解析プロセスにさらなる効率化と自動化をもたらすと期待されています。例えば、モデルに基づくシミュレーションやリアルワールドデータ（RWD）の活用において、Rの柔軟性は極めて重要です。 admiralは、これらの高度な解析のための高品質なデータ基盤を提供する上で、ますますその重要性を増していくでしょう。

よくある質問（FAQ）

Q1: admiralパッケージは、SASに完全に置き換わるものですか？

A1: 現時点では、完全に置き換わるというよりは、SASとRを組み合わせたハイブリッドなワークフローが主流です。特に、ADaMデータセットの作成においては、admiralが高い生産性と品質を提供しますが、既存のSAS資産や規制要件、社内のスキルセットに応じて、最適なツールを選択することが重要です。長期的な視点では、Rへの移行が加速すると考えられます。

Q2: admiralで作成したADaMデータセットは、規制当局に提出できますか？

A2: はい、可能です。admiralはCDISC ADaM IGに準拠したデータセット作成を支援するように設計されており、適切なバリデーション（検証）プロセスとドキュメンテーション（文書化）があれば、規制当局に提出できます。FDAはR言語で作成された解析プログラムやデータセットを受け入れる方針を示しており、今後、Rベースの提出事例は増加していくでしょう。

Q3: admiralを学ぶ上で、どのような前提知識が必要ですか？

A3: admiralはtidyverseのエコシステム上に構築されているため、dplyrやmagrittr（パイプ演算子）といったtidyverseの基本的な知識があるとスムーズに学習できます。また、CDISC ADaMの概念や臨床試験データの構造に関する理解も不可欠です。Rの基本から学びたい場合は、R for Data Scienceなどの入門書から始めることをお勧めします。

Q4: SASマクロとadmiralの主な違いは何ですか？

A4: SASマクロは柔軟性が高い反面、可読性やメンテナンス性が課題となることがあります。一方、admiralはtidyverseの関数型プログラミングの思想に基づき、モジュール化された関数を提供することで、可読性と再利用性を大幅に向上させています。また、admiralはCDISC ADaMのベストプラクティスを内包しているため、より標準化されたコードを記述しやすいという利点があります。
