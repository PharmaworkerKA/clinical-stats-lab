admiral Rで臨床試験ADaMを効率化！SDTMからAIプログラミング

Photo from Unsplash

皆さん、こんにちは！臨床統計ラボへようこそ。臨床試験のデータ解析において、ADaM（Analysis Data Model）データセットの作成は非常に重要な工程でありながら、多くの時間と労力を要する作業ですよね。特に、SDTM（Study Data Tabulation Model）からADaMへの変換作業は、臨床統計担当者やSASプログラマーにとって大きな負担となることがあります。

しかし、近年Rコミュニティで開発が進むRパッケージ「admiral」は、この課題に対する強力なソリューションを提供しています。admiralは、臨床試験のADaMデータセット作成を効率化するために特化して設計されており、SDTMデータを活用して標準化された分析データセットを迅速に生成することを可能にします。本記事では、このadmiral Rパッケージの基本から、具体的なコード例、さらにはAIプログラミングとの連携によるさらなる効率化の可能性まで、臨床統計のエキスパートとして詳しく解説していきます。

結論から言うと

admiral Rパッケージは、臨床試験におけるADaMデータセットの作成を劇的に効率化します。SDTMからADaMへの変換を標準化された関数群で支援し、再現性の高いプログラミングを可能にすることで、解析担当者の負担を大幅に軽減します。特に、tidyverseエコシステムとの親和性が高く、現代のデータサイエンス手法と組み合わせることで、データ解析の自動化・効率化を加速させるでしょう。

admiralとは？

admiralは、Rパッケージの一つで、臨床試験におけるADaMデータセット（解析用データセット）の作成に特化したオープンソースプロジェクトです。CDISC（Clinical Data Interchange Standards Consortium）のADaMIG（ADaM Implementation Guide）に準拠したデータセットを効率的かつ標準的に作成できるよう設計されています。

従来のADaMデータセット作成は、SASプログラミングが主流でしたが、admiralはR言語の強み、特にtidyverse（データ操作のためのRパッケージ群）の概念を取り入れることで、より直感的で記述性の高いコードでのデータ変換を可能にしました。SDTM（Study Data Tabulation Model）データからADaMデータセットへ変換する際の様々な導出（例: 年齢計算、ベースライン値の特定、期間算出など）を、検証済みの関数として提供しています。

admiralの主要な特徴

  • CDISC ADaMIG準拠: ADaMデータセットの構造や変数導出ルールに則った関数を提供し、規制当局への提出要件を満たしやすい設計です。
  • Tidyverseベース: dplyrのパイプ演算子（%>%）と親和性が高く、データフローが追いやすいコードを書くことができます。
  • 再現性と標準化: 共通のロジックを関数として提供することで、プロジェクト間でのコードの標準化を促進し、解析の再現性を高めます。
  • コミュニティ駆動: グローバルなオープンソースコミュニティによって開発・維持されており、活発な議論と機能改善が進んでいます。

統計的背景・理論

ADaMデータセットは、臨床試験の解析において非常に重要な役割を担います。その主要な目的は、SDTMという報告用の標準データセットから、統計解析に適した形式へとデータを変換し、トレーサビリティ（追跡可能性）を確保することにあります。

ADaMの重要性とトレーサビリティ

ADaMは、規制当局（FDAやPMDAなど）への申請において、統計解析の結果を裏付ける重要な証拠となります。ADaMIGでは、以下の3つの主要な原則が求められています。

  1. Traceability（追跡可能性）: SDTMデータからADaMデータ、そして最終的な解析結果（Tables, Listings, Figures; TLFs）まで、データの変換経路が明確に追跡できること。これは、解析の透明性と信頼性を保証するために不可欠です。
  2. Clarity（明確性）: ADaMデータセットの構造や変数の導出ロジックが明確であること。外部のレビューアが容易に理解できる必要があります。
  3. Integrity（完全性）: データ変換プロセスにおいて、データの完全性が維持されていること。不適切なデータの変更や削除がないことを意味します。

admiralは、これらの原則、特にトレーサビリティをRの関数を通じてサポートします。例えば、derive_vars_merged()のような関数は、どのデータセットからどの変数をマージしたかをコード上で明確にし、また、derive_vars_extreme_flag()のような関数は、ベースライン値などの導出ロジックを標準化することで、解析担当者が複雑な導出ルールを自力で実装する手間を省きつつ、一貫したアプローチを提供します。

SAS/Rコード実装

ここでは、admiral Rパッケージを使って、SDTMデータからADaM-ADSL（被験者レベル分析データセット）を作成する具体的なコード例をご紹介します。ADaMデータセットの中でも、ADSLは最も基本的なもので、他のADaMデータセットのベースとなるため、その作成は非常に重要です。

1. 必要なRパッケージのロードとサンプルSDTMデータの準備

まず、tidyverse（データ操作の基本）、admiral（ADaM作成）、rtables（解析結果の表作成）をロードします。次に、解析の基礎となるSDTMデータセット（ここではDM: 被験者デモグラフィック、VS: バイタルサイン を簡略化して作成）を用意します。

# 必要なRパッケージのロード
library(tidyverse) # データ操作の基本
library(admiral)   # ADaMデータセット作成の中核パッケージ
library(admiraldev) # admiral開発用のヘルパー関数。ここでは日付変換に使用
library(rtables)   # 解析結果の表（Tables）作成パッケージ

# サンプルSDTM DMデータセットの作成
# DM (Demographics) データセットは被験者のデモグラフィック情報を含みます
dm <- tibble::tribble(

  "CDISCPILOT", "CDISCPILOT-003", "003", "S002", "1988-11-01", "M", "ARM A", "Treatment A", "ARM A", "Treatment A", NA, NA,
  "CDISCPILOT", "CDISCPILOT-004", "004", "S002", "1965-06-25", "F", "ARM C", "Placebo", "ARM C", "Placebo", "2023-01-10", "Y",
  "CDISCPILOT", "CDISCPILOT-005", "005", "S003", "1992-09-10", "M", "ARM B", "Treatment B", "ARM B", "Treatment B", NA, NA

dm <- convert_dtc_to_dt(dm, vars(BRTHDTC, DTHDTC, RFSTDTC, RFENDTC))
vs <- convert_dtc_to_dt(vs, vars(VSDTC))

2. ADSL (Subject-Level Analysis Dataset) の作成

次に、admiralの関数を使ってADSLデータセットを作成します。ADSLには、被験者のデモグラフィック情報、治験薬投与情報、主要なイベント（死亡など）に関する情報が含まれます。

# ADSLデータセットの作成
adsl <- dm %>%
  # 被験者レベルの参照開始日と終了日をderive (導出)
  # derive_vars_dt() はDTC形式の日付文字列からDT形式の日付変数を導出します。
  derive_vars_dt(
    dtc = RFSTDTC,
    new_vars_prefix = "RFST" # 新しい変数名の接頭辞
  ) %>%
  derive_vars_dt(
    dtc = RFENDTC,
    new_vars_prefix = "RFEN"
  ) %>%
  # 年齢を計算 (ADSLでは年齢 (AGE) が必要)
  # derive_vars_age() は生年月日と基準日から年齢を計算します。
  derive_vars_age(
    new_var = AGE,
    date_var = RFSTDTC,     # 参照開始日時点の年齢を計算
    birth_date_var = BRTHDT,
    unit = "years",
    type = "continuous"
  ) %>%
  # 死亡フラグ (DTHFL) をderive
  # derive_vars_event_flag() は特定のイベントに基づいてフラグ変数を導出します。
  derive_vars_event_flag(
    dataset_adsl = .,       # ADSLデータセットをパイプで渡す
    dataset_source = dm,    # ソースデータセットとしてDMを使用
    filter_source = DTHFL == "Y", # DMのDTHFLが'Y'の場合に死亡と判断
    new_var = DTHFL
  ) %>%
  # 投与開始日 (TRTSDT) を仮で参照開始日と同じとする
  mutate(TRTSDT = RFSTDTC) %>%
  # 治療期間に関する変数 (TRT01P: 計画治療, TRT01A: 実際の治療) を導出
  # ADaMIGに沿って適切なラベルやコードを割り当てるのが一般的です。
  mutate(TRT01P = ARM, TRT01A = ACTARM) %>%
  # 変数ラベルの追加
  # set_variable_labels() はADaM変数のラベルを設定します。
  set_variable_labels(
    USUBJID = "Unique Subject Identifier",
    BRTHDT = "Date of Birth",
    AGE = "Age (Years)",
    SEX = "Sex",
    ARM = "Planned Arm",
    ACTARM = "Actual Arm",
    DTHFL = "Death Flag",
    TRT01P = "Planned Treatment",
    TRT01A = "Actual Treatment"
  )

# 作成されたADSLデータセットの表示
print(adsl)

3. rtablesによる解析結果テーブルの生成

rtablesパッケージは、臨床試験の報告書で求められるような高品質なテーブルをRで生成するために設計されています。ここでは、作成したADSLデータセットから被験者の特性（年齢、性別）を要約した簡単なテーブルを作成する例を示します。

# rtablesを使ってADSLから簡単な被験者特性サマリーテーブルを作成

# テーブルレイアウトの定義
table <- rtables::build_table(
  basic_table() %>% # 基本的なテーブル構造を開始
    split_cols_by("TRT01A") %>% # 実際の治療群 (TRT01A) で列を分割
    add_overall_col("Overall") %>% # 全体列を追加
    # 年齢 (AGE) と性別 (SEX) の変数を解析
    analyze(vars = c("AGE", "SEX"),
            fns = list(
              AGE = rtables::r_stats_summary(mean = mean, sd = sd, digits = 1), # 年齢は平均と標準偏差
              SEX = rtables::r_count_pct(denom = "N_col") # 性別は度数と割合
            )) %>%
    append_topleft(c("Characteristic", "N")) %>% # 左上に項目名とN数を追加
    # N=() の形式でN数を表示
    append_topleft(paste0("N=", formatC(ncol_source_value(.), big.mark = ","))),
  df = adsl # 作成したADSLデータフレームをテーブルのデータソースとして渡す
)

# 生成されたテーブルの表示
print(table)

このコード例は、admiralがSDTMからADaMへの変換をいかに効率的かつ標準的に行うかを具体的に示しています。また、rtablesを組み合わせることで、ADaM作成から解析結果の可視化までの一連のワークフローをRでシームレスに実現できることがお分かりいただけたかと思います。

実務での活用場面

admiralやその他のRパッケージは、臨床試験の様々なフェーズでその真価を発揮します。

  1. プロトコル（治験計画書）策定段階でのシミュレーション: 計画段階で仮のSDTMデータを生成し、admiralを使ってADaM作成のロジックを事前に検討・検証できます。これにより、解析計画の実現可能性を評価し、プロトコルや統計解析計画書（SAP）の記述をより具体的にできます。
  2. データクリーニングと検証: SDTMデータが完成した後、admiralを使ってADaMを生成し、その導出結果を検証することで、データの問題点や逸脱を早期に発見できます。
  3. 統計解析計画書（SAP）の実行: SAPに記述された解析手法に従い、必要なADaMデータセットをadmiralで一貫して作成します。これにより、統計解析担当者はデータの準備作業に費やす時間を削減し、本来の解析業務に集中できます。
  4. 治験薬承認申請（NDA/MAA）: 規制当局への提出データとして、CDISC標準に準拠したADaMデータセットは必須です。admiralは、この要件を満たす高品質なデータセット作成を支援します。
  5. 変更管理と再現性: 治験中にプロトコルが変更された場合でも、admiralのコードベースはモジュール化されているため、変更箇所の特定と修正が容易です。これにより、一貫性と再現性を保ちながら、効率的にデータセットを更新できます。

海外の最新動向

R言語とadmiralのようなRパッケージは、海外の臨床試験コミュニティ、特にPSI（Statisticians in the Pharmaceutical Industry）、ASA Biopharmaceutical Section、PHUSE（Pharmaceutical Users Software Exchange）などで活発に議論されています。

R in Clinical Trialsの推進

数年前まではSASが圧倒的多数を占めていましたが、近年はRの利用が急速に拡大しています。特に、PHUSEやR Consortiumが主催する「R in Clinical Trials」プロジェクトは、Rを臨床試験の環境で利用するためのガイドライン策定やツールの開発を推進しています。admiralもこの動きの一環として開発され、公式なCDISC標準への対応や規制当局への提出における課題を解決するための主要なツールの一つと認識されています。

AIプログラミングとadmiralの連携

「AIプログラミング」というキーワードは、今後の臨床試験のデータ解析を大きく変える可能性を秘めています。具体的には、以下のような連携が期待されています。

  1. コード生成の自動化: 大規模言語モデル（LLM）のようなAIが、自然言語で書かれたADaM導出ルールや解析計画を解釈し、admiralの関数を使ったRコードを自動生成する。これにより、プログラマーの作業負荷を軽減し、初学者でも高品質なコードを書けるようになります。
  2. 品質管理（QC）の強化: AIがadmiralで生成されたADaMデータセットと既存のCDISC標準、または過去の承認済みデータセットを比較し、潜在的なエラーや逸脱を自動的に検出する。これにより、QCプロセスが大幅に効率化されます。
  3. データフローの最適化: 複雑なSDTMからADaMへの変換パスにおいて、AIが最も効率的なデータ処理フローを提案し、最適なadmiral関数の組み合わせを推奨する。

これらの動向は、臨床試験のデータ解析プロセス全体を再構築し、より迅速かつ高品質な医薬品開発に貢献するでしょう。admiralは、その中心的なツールとして進化を続けています。

よくある質問（FAQ）

Q1: admiralを使う最大のメリットは何ですか？

A1: admiralの最大のメリットは、臨床試験のADaMデータセット作成をCDISC標準に準拠した形で効率化し、かつ再現性を高める点にあります。検証済みの関数を使うことで、プログラミングの労力を削減し、品質管理を容易にします。

Q2: SASからR (admiral) への移行は難しいですか？

A2: SASの経験がある方がRのtidyverseの考え方に慣れるまでには学習曲線がありますが、admiralはtidyverseの記法をベースにしているため、比較的スムーズに移行できます。特に、既存のSASコードをRで書き換える際の強力なツールとなります。{{internal_link:SASプログラマーのためのR入門}}のような記事も参考にしてください。

Q3: admiralは本当にCDISC標準に完全に準拠していますか？

A3: はい、admiralはCDISC ADaMIG（ADaM Implementation Guide）に沿って設計・開発されており、CDISC標準への準拠を強く意識しています。ただし、個別の臨床試験の要件や規制当局の最新ガイダンスに従うためには、利用者が適切な判断と調整を行う必要があります。{{internal_link:CDISCとは？基本と役割}}で詳細をご確認ください。

Q4: AIプログラミングは具体的にADaM作成にどう活用できるのですか？

A4: AIプログラミング、特に大規模言語モデル（LLM）の活用は、ADaM導出ロジックの自然言語記述からのRコード自動生成、既存コードのレビューと最適化、品質管理における異常検知などに役立ちます。これにより、プログラマーはより戦略的なタスクに集中できるようになり、効率化が図られます。
